## 1. 项目概述
### 1.1 项目背景
业务人员在进行日常数据分析时，严重依赖研发人员编写SQL或制作固定报表，导致需求响应周期长、灵活性差。本项目旨在构建一款基于Multi-Agent（多智能体）协同架构的AI数据助手，让不懂代码的业务人员通过“自然语言对话”的方式，安全、高效地获取、分析业务数据并生成报表。
### 1.2 核心目标
*   **零门槛交互：** 纯自然语言转SQL（Text-to-SQL），屏蔽底层主流数据库差异。
*   **安全可控：** 核心原则“读数据随意，改数据必批”，引入强制人工干预机制。
*   **深度洞察：** 从“取数据”升级为“分析数据+生成可视化报表”。
*   **无缝集成：** 提供标准化API，支持独立部署与快速嵌入已有业务系统。
---
## 2. 系统功能架构设计
系统采用“前后端分离 + 多层解耦”的架构设计：
*   **表现层：** Vue前端（独立Chat界面） / 第三方业务系统（通过API接入）。
*   **接口网关层：** FastAPI（负责鉴权、限流、SSE流式推送、审批接口）。
*   **Agent编排层：** 基于LangGraph 1.x构建的有状态工作流引擎（负责节点路由、状态记忆、人工打断）。
*   **多智能体协同层：** 参考Hermes工程架构，按职责划分专职Agent。
*   **能力基座层：** Langchain 1.x（工具调用/SQL工具链）、deepAgents 0.5.x（底层LLM调度与结构化输出增强）、RAG检索链。
*   **数据存储层：** 业务数据库集群、Milvus向量数据库（存储操作规范/元数据）。
---
## 3. Multi-Agent 协同架构设计 (参考 Hermes Engineering)
摒弃单体大而全的Agent，采用**控制平面+ 数据平面 + 专家Worker**的Hermes架构模式：
1.  **Orchestrator Agent (控制枢纽):** 
    *   负责接收用户意图，进行任务拆解。
    *   动态路由到合适的Worker Agent。
    *   管理对话上下文和LangGraph的全局State。
2.  **RAG Agent (规范检索专家):** 
    *   监听用户输入，判断是否涉及“如何使用”、“指标定义”等问题。
    *   从Milvus中检索《系统操作规范》、《数据字典》并返回。
3.  **Schema Agent (元数据专家):** 
    *   根据用户意图，动态从各业务库加载相关的表结构、字段注释。
    *   过滤掉无关表，降低Text-to-SQL的Token消耗。
4.  **SQL Coder Agent (编码专家):** 
    *   接收精简后的Schema和用户问题，生成目标方言SQL（MySQL/PG/Oracle等）。
    *   具备SQL自校正能力（如果执行报错，能将错误信息喂给LLM重新生成）。
5.  **Security Agent (安全审计专家):** 
    *   **核心职责：** 使用LLM或正则对生成的SQL进行分类分级。
    *   判定规则：`SELECT` -> 放行；`INSERT/UPDATE/DELETE/DROP/TRUNCATE` -> 标记为高危，强制触发Human-in-the-loop（人工审批流）。
6.  **Analyst Agent (数据分析师):** 
    *   接收SQL执行结果集。
    *   进行基础统计分析（同比、环比、极值、异常点识别），生成自然语言分析洞察。
7.  **Reporter Agent (报表生成专家):** 
    *   根据数据特征，推荐图表类型（折线图、饼图、柱状图）。
    *   输出标准的Echarts JSON配置或生成Excel文件流。

## 4. Agent 核心工作流
基于LangGraph 1.x实现，支持状态保持与中断恢复：
```text
[用户输入] 
   │
   ▼
[1. 意图路由] ----------------------------------------> (如果是提问/求助) 
   │                                                        │
   │ (如果是查数据/写数据)                                     ▼
   ▼                                                   [RAG Agent 检索]
[2. Schema Agent 获取表结构]                                  │
   │                                                        │
   ▼                                                        │
[3. SQL Coder Agent 生成SQL] <-------------------------------┘
   │
   ▼
[4. Security Agent 审计] 
   │
   ├── (安全: SELECT) ──┐
   │                    ▼
   │             [5. 执行SQL获取数据]
   │                    │
   │                    ▼
   │             [6. Analyst Agent 数据分析]
   │                    │
   │                    ▼
   │             [7. Reporter Agent 图表/报表生成]
   │                    │
   ├── (高危: DML/DDL) ─┼──> [8. 挂起任务, 通过FastAPI推送审批流给管理员]
   │                    │         │
   │                    │         ├── (管理员驳回) -> 返回拒绝结果给用户
   │                    │         │
   │                    │         └── (管理员通过) -> 进入 [5. 执行SQL获取数据]
   │                    │
   ▼                    ▼
[9. 统一结果组装与前端流式输出]
```

## 5. 详细功能需求
### 5.1 核心基础功能
| 功能模块 | 需求描述 | 验收标准 |
| :--- | :--- | :--- |
| **方言自适应** | 支持MySQL, PostgreSQL, SQL Server, Oracle。 | 根据配置的数据源类型，自动生成对应方言的语法（如PG的`::date`，Oracle的`to_date`）。 |
| **Text-to-SQL** | 用户输入自然语言，生成高准确率SQL。 | 复杂表关联（JOIN）、分组聚合（GROUP BY）、时间范围查询准确率>85%。 |
| **SQL自纠错** | SQL执行失败时的自我修复。 | 拦截语法错误或字段不存在的报错，最多重试2次，避免死循环。 |
### 5.2 便利性与高阶功能
| 功能模块 | 需求描述 | 验收标准 |
| :--- | :--- | :--- |
| **智能数据分析** | 不仅返回数据，还返回结论。 | 例如：“查询上月销售额”，不仅返回数字，还自动计算并回答“环比增长了15%”。 |
| **动态可视化** | 根据数据自动生成图表。 | 前端能够直接渲染Agent返回的Echarts配置，支持一键切换图表类型。 |
| **报表导出** | 将查询结果保存为文件。 | 支持当前对话数据一键导出为Excel或CSV格式。 |
| **Skill集成** | 预留外部工具调用能力。 | 支持调用外部API（如发送企业微信通知、调用第三方天气API辅助分析）。 |
### 5.3 RAG 操作规范功能
| 功能模块 | 需求描述 | 验收标准 |
| :--- | :--- | :--- |
| **规范知识库** | 将《指标字典》、《系统使用手册》向量化存入Milvus。 | 支持文档上传、自动切片、向量化入库。 |
| **主动检索** | 当用户问题模糊或询问操作方法时触发。 | 问“怎么用”时，不生成SQL，而是从Milvus检索出操作指引返回。 |
| **隐式增强** | 在Text-to-SQL前，检索相关指标的准确定义。 | 例如用户说“GMV”，RAG能检索到GMV的计算公式（订单金额-退款），辅助SQL生成。 |
### 5.4 安全与人工干预机制
| 功能模块 | 需求描述 | 验收标准 |
| :--- | :--- | :--- |
| **高危操作拦截** | 绝对禁止直接执行增删改操作。 | 正则+LLM双重校验，识别到`UPDATE/DELETE/INSERT/DROP/ALTER`等关键词立即拦截。 |
| **审批工作流** | 触发拦截后，冻结当前Graph状态。 | 通过WebSocket/SSE通知前端/业务系统有待审批任务；管理员通过FastAPI接口`/approve/{task_id}`回调，Graph恢复执行。 |
| **数据脱敏** | 返回结果中的敏感字段打码。 | 手机号、身份证号等在SQL执行后、返回前端前进行正则替换脱敏。 |

## 6. 系统接口设计
提供基于FastAPI的标准RESTful API，方便业务系统集成：
| 接口路径 | 方法 | 描述 |
| :--- | :--- | :--- |
| `/api/v1/chat/completions` | `POST` | **核心对话接口**，返回SSE流式数据（包含文本、图表配置、中间态）。 |
| `/api/v1/chat/approve` | `POST` | **人工审批接口**，接收`task_id`和`action` (approve/reject)。 |
| `/api/v1/chat/stop` | `POST` | 中断当前Agent执行。 |
| `/api/v1/datasource/list` | `GET` | 获取当前用户有权限的数据源列表。 |
| `/api/v1/export/excel` | `POST` | 根据对话历史中的数据集ID，触发异步导出，返回下载链接。 |

## 7. 技术栈与部署架构
### 7.1 技术选型明细
*   **后端核心：** Python 3.10+
*   **LLM编排框架：** LangChain 1.x (工具封装、输出解析器)
*   **Agent工作流：** LangGraph 1.x (状态图构建、Human-in-the-loop打断机制)
*   **底层Agent引擎：** deepAgents 0.5.x (用于增强各个子Worker的结构化推理能力与Prompt管理)
*   **Web框架：** FastAPI (异步高并发、原生SSE支持)
*   **前端：** Vue 3 + Vite + Element Plus (UI组件) + Echarts (图表)
*   **向量数据库：** Milvus 2.x (存储RAG知识库)
*   **业务数据库驱动：** SQLAlchemy (统一ORM与多方言适配)
*   **LLM模型：** 私有化部署Qwen或GLM系列大模型（需具备较强的代码与指令遵循能力）
### 7.2 部署架构
*   **独立部署模式：** 前端打包为静态文件由Nginx托管，后端采用Docker容器化部署。
*   **集成部署模式：** 业务系统仅调用FastAPI接口，无需部署Vue前端。
*   **基础设施：** Kubernetes (可选) / Docker Compose (开发测试)，Redis用于存储Graph中断状态的快照。

## 9. 非功能性需求
1.  **性能要求：** 
    *   简单查询（单表）端到端响应时间 < 5秒。
    *   复杂查询（多表JOIN）端到端响应时间 < 15秒。
    *   接口支持SSE流式输出，首字响应时间 < 1秒。
2.  **可用性：** 系统可用性达到 99.9%，LLM调用失败时需有优雅降级提示。
3.  **可扩展性：** 新增一种数据库类型，只需修改配置文件和增加对应SQL方言模板，无需改动核心Agent逻辑。
4.  **并发限制：** 通过FastAPI配置单用户并发限流，防止LLM资源被耗尽。
