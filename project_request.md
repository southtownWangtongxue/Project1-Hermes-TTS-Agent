## 1. 项目概述
### 1.1 项目背景
企业内部大量业务数据存储在关系型数据库中，业务人员（运营、销售、财务等）不具备 SQL 编写能力，数据分析需求长期依赖技术部门取数，响应慢、沟通成本高。为降低数据使用门槛，亟需一款以自然语言驱动的数据库查询与分析助手。

### 1.2 项目目标
构建一个基于大语言模型（LLM）的 AI Agent，实现：
- **Text-to-SQL**：自然语言转 SQL，支持多轮交互与上下文理解。
- **自助数据分析**：将查询结果自动总结、可视化、生成报表。
- **安全可控**：对增删改等敏感操作必须人工审批。
- **系统集成**：通过 FastAPI 接口嵌入已有业务系统。
- **独立部署**：提供前端聊天界面，开箱即用。
- **操作规范辅助**：内置 RAG 知识库，在必要时检索并告知用户正确操作方法。

## 3. 核心工作流 (LangGraph 状态图)
以下为一次用户查询的完整流转过程，图中标出关键决策点与人工干预位置。

```mermaid
graph TD
    A[用户输入自然语言] --> B{意图分类}
    B -->|数据查询/分析| C[检索RAG操作规范<br/>+ 相关表结构]
    B -->|操作类/DDL/DML| D[进入敏感操作审批流程]
    C --> E[生成SQL语句]
    E --> F[SQL语法校验与安全检测]
    F -->|不通过| E
    F -->|通过| G[执行查询(SELECT)]
    G --> H[结果分析 & 自然语言总结]
    H --> I{用户要求图表/报表?}
    I -->|是| J[生成可视化图表/报表文件]
    I -->|否| K[仅文本回复]
    D --> L[提交人工审批界面]
    L -->|审批通过| M[执行写操作并记录日志]
    L -->|驳回| N[通知用户并建议修改]
    J --> O[返回结果给用户]
    K --> O
    M --> O
    N --> O
```

**关键干预点说明：**
- **敏感操作判定**：任何非 `SELECT` 类语句，或涉及高危表/字段的操作，均进入审批。
- **RAG 介入时机**：用户表述模糊、表字段不确定、操作涉及业务规范时，主动检索并给出提示。

## 4. 详细功能需求
### 4.1 基础功能：Text-to-SQL
- **多轮对话**：记忆上下文，指代消解（如“上个月的”“那个产品”）。
- **多数据库支持**：动态适配 MySQL、PostgreSQL、Oracle、SQL Server 等 SQL 方言。
- **表结构感知**：自动读取数据库元数据（表名、字段、注释），并向量化存入 Milvus。
- **模糊匹配**：用户字段名输入错误或使用近义词时，通过向量相似度推荐正确的列。
- **查询优化提示**：若生成的 SQL 明显低效（如全表扫描），主动提醒并提供优化建议。

### 4.2 必备扩展功能
| 功能 | 描述 |
|------|------|
| **数据分析** | 对查询结果自动进行聚合统计、趋势描述、异常检测，输出自然语言解读。 |
| **报表生成** | 支持将多步查询结果组合成表格、柱状图、折线图等，导出为 Excel/PDF/图片。 |
| **图表可视化** | 基于 ECharts 在聊天界面内联展示图形，支持缩放、数据点查看。 |
| **定时任务/订阅** | 允许用户设置“每日销售报告”等，Agent 定时执行并推送。 |
| **多渠道通知** | 报表完成后通过邮件、企业微信、钉钉等通知用户。 |

### 4.3 便利性功能
- **快捷提问模板**：预置常见查询（如“本月各区域销售额TOP10”）。
- **个人查询历史**：可搜索、复用、收藏历史问题。
- **自然语言→操作教程**：当用户问“如何导出上月考勤？”时，Agent 不仅生成SQL，同时从 RAG 中检索相关操作规范并展示。
- **多 Agent/技能集成**：
  - **SQL 优化 Agent**：专门负责改写低效 SQL。
  - **数据解释 Agent**：调用外部统计模型或 Python 代码解释器。
  - **第三方系统 Agent**：通过 API 触发 ERP/CRM 中的流程。

### 4.4 敏感操作与人工审批
**判定规则（可配置）：**
- 操作类型：`INSERT`、`UPDATE`、`DELETE`、`DROP`、`ALTER`、`TRUNCATE` 等。
- 目标表：标记为“高危”的业务表（如财务、薪酬）。
- 影响行数：`UPDATE/DELETE` 不带 `WHERE` 或预计影响>N 行。

**审批流程：**
1. Agent 生成操作预览（SQL 语句、预计影响行数、风险等级）。
2. 提交至审批队列，通过前端界面或 API 通知指定审批人。
3. 审批人可查看详情、批准/驳回，并附言。
4. 批准后 Agent 在限定时间内执行，并保留完整审计日志。

## 5. 系统接口设计 (FastAPI)
### 5.1 核心接口
```yaml
POST /api/v1/chat
  描述: 会话式查询
  请求体:
    session_id: string
    message: string
    user_id: string
  响应:
    reply: string
    sql: string (optional)
    data: object (optional, 查询结果)
    chart: object (optional, 图表配置)
    references: [操作规范引用列表]

GET /api/v1/sessions/{session_id}
  描述: 获取历史对话

POST /api/v1/approval/requests
  描述: 审批申请 (由Agent自动创建)

POST /api/v1/approval/{id}/review
  描述: 审批操作 (通过/拒绝)
```

### 5.2 集成方式
- 现有业务系统通过 HTTP 调用 `/api/v1/chat`，传入用户问题，获取结构化回复，直接在前端渲染。
- 支持 API Key + HMAC 鉴权，防止未授权访问。

## 6. RAG 功能与操作规范库
### 6.1 知识库内容
- **数据库字典**：表名、字段名、类型、业务含义、关联关系。
- **SQL 编写规范**：命名规则、索引使用建议、常见误区。
- **业务操作手册**：如“如何申请数据导出”“月度结算查询标准流程”。
- **历史优质对话**：经审核的高频查询示例，提升相似问题的准确性。

### 6.2 RAG 工作流
1. 用户问题向量化 → Milvus 相似度检索。
2. 返回 Top-K 相关知识片段。
3. 将片段织入 System Prompt 或作为 Tool 输出，指导 Agent 回答。
4. 若知识库中明确存在标准操作方法，Agent 须优先引用并以可点击链接展示。
